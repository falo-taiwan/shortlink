短網址 v1 說明頁

這一版不是短網址服務，而是先把 shortlink 的資料結構、維護流程與 GitHub Pages routing 路線定下來。

你可以把它理解成：`links.json` 是資料表，`404.html` 是 router，`/go/<slug>` 是對外入口，而 `shortlink JSON editor` 是人實際會操作的維護工具。

shortlink v1 links.json GitHub Pages ready

這是什麼工具

這套 shortlink v1 不是重系統，也沒有後台。它先用最小做法，把短網址維護拆成兩層：

1. `links.json`：負責存放 slug 與 target URL。
2. `404.html`：負責接管 `/go/<slug>` 並查表 redirect。

這樣的好處是簡單、穩定、好教，也很適合放在 GitHub Pages 上先驗證整體路線。

v1 工作方式

1. 在 shortlink editor 中讀取或上傳 `links.json`。
2. 新增、修改、刪除 slug 與 target URL。
3. 匯出新的 `links.json`。
4. 人工覆蓋到主站 repo root。
5. 之後由 `404.html` 讀取 `links.json`，處理 `/go/<slug>`。

links.json 範例格式

v1 先固定成最簡字串對字串，不先做 title、note、status、tags。這是故意的，因為第一步要先把資料面與 router 介面穩住。

{
  "123": "https://www.google.com/",
  "demo": "https://force.formosa-ai.com",
  "force": "https://force.formosa-ai.com/about/",
  "qrcode": "https://falo-taiwan.github.io/tools/qrcode/falo-qrcode-v102.html",
  "shortlink": "https://falo-taiwan.github.io/tools/shortlink/falo-shortlink-json-editor-v101.html"
}

建議目錄結構

主站 repo 是 `falo-taiwan.github.io`。v1 建議這樣放，之後人工更新時最直覺：

/
├─ 404.html
├─ links.json
├─ /go/
│   └─ （未來 shortlink 入口路徑）
├─ /tools/
│   ├─ /qrcode/
│   │   ├─ falo-qrcode-v102.html
│   │   └─ README.md
│   └─ /shortlink/
│       ├─ falo-shortlink-json-editor-v101.html
│       ├─ falo-shortlink-guide-v101.html
│       └─ README.md

重點是 `links.json` 與 `404.html` 都放在 repo root，這樣 router 後續就能直接吃同一份資料。

人工更新流程

1. 打開 `falo-shortlink-json-editor-v101.html`。
2. 先嘗試自動讀取預設 `links.json`。
3. 如果失敗，就手動上傳既有 `links.json`。
4. 完成新增、修改、刪除。
5. 匯出新的 `links.json`。
6. 人工覆蓋主站 repo root 的 `links.json`。
7. 重新部署後，新的 `/go/<slug>` 對應就會生效。

未來如何升級到 GAS

v2 再把資料來源升級成 `GAS + JSON fallback`。也就是：

1. 正式維護來源改成 GAS。
2. `404.html` 先嘗試拿 GAS 指令。
3. 若 GAS timeout、格式錯、沒有 target，就 fallback 改吃 root 的 `links.json`。

這樣 v1 的 `links.json` 不會浪費，而會變成後續系統的保底層。

這一版的限制

1. 沒有自動寫回 GitHub，仍需人工覆蓋 `links.json`。
2. 沒有 GAS、Google Sheet、登入與權限控管。
3. 沒有批量產生 `/go/<slug>/index.html` 靜態頁。
4. 沒有多語系與 SEO 強化，這一版先專注在資料面。
